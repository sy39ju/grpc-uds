# grpcuds from plain C (BLE scanner)

The same BLE scanner contract the C++ example (`../ble`) implements,
written in **plain C** ‚Äî two `.c` files, no C++ anywhere. `grpcuds.h` +
`libgrpcuds_ffi.a` are the whole runtime; messages are nanopb and the
service stubs are generated by `protoc-gen-grpcudspp --grpcudspp_opt=c`.
Stock gRPC clients (grpcurl, tonic, grpc++) can call this server unchanged.

```sh
# in the repo: ./build.sh first (the .a), then
cmake -S . -B build && cmake --build build
./run_demo.sh build        # BLE scan round-trip -> "example: OK"
```

What each file shows:

- **`server.c`** ‚Äî fill the generated `ble_BleScanner_service` table
  (Init / StartLeScan / ScanResultStream / StopLeScan / ReadCharacteristic)
  and `ble_BleScanner_register()` it; a simulated radio answers. Streaming
  runs **on a producer thread**: `ScanResultStream` spawns it and returns
  immediately (deferred), and the thread pushes adverts with
  `ble_BleScanner_ScanResultStream_send`/`_finish` ‚Äî which are **thread-safe**
  (the C ABI routes off-I/O-thread writes through the outbound mailbox). The
  I/O thread calls `grpcuds_mailbox_register_io_thread`, polls the mailbox
  wakeup fd, and `grpcuds_mailbox_drain`s ‚Äî same shape as the C++ example, with
  the mailbox now in the shared C ABI (`docs/THREADING.md`).
- **`client.c`** ‚Äî `grpcuds_client_connect_wait`, then the generated typed
  wrappers: `ble_BleScanner_Init`, `..._StartLeScan`,
  `..._ScanResultStream_start`/`_next`, `..._StopLeScan`,
  `..._ReadCharacteristic`.
- **Health checking** ‚Äî the server `grpcuds_health_register`s the standard
  `grpc.health.v1.Health` service (built into the C ABI); the client runs a
  Check (`SERVING`, and `NOT_FOUND` for an unknown name) and a Watch probe
  before the scan. Stock probers work unmodified.
- Both register a stderr **log sink** (`grpcuds_set_log_callback`) ‚Äî the
  library is silent without one.

## Codegen (C)

Two passes, both run by CMake ‚Äî here is what it runs (proto =
`sdk/proto/ble.proto` in the bundle, `../ble/proto/ble.proto` in the repo;
`plugin` = the `protoc-gen-grpcudspp` from `../../build.sh`):

```sh
# 1. nanopb ‚Äî message structs (ble.pb.{h,c})
nanopb_generator -D gen -I proto -f proto/ble.options proto/ble.proto

# 2. protoc-gen-grpcudspp, C MODE ‚Äî service table + typed client wrappers
protoc --plugin=protoc-gen-grpcudspp="$plugin" \
       --grpcudspp_out=gen --grpcudspp_opt=c \
       --proto_path=proto proto/ble.proto
# -> gen/ble.grpcuds.h  (ble_BleScanner_service table; client wrappers,
#                        static inline) and gen/ble.grpcuds.c (server
#                        trampolines + ble_BleScanner_register)
```

The same `ble.proto` the C++ example uses. Service codegen is optional ‚Äî
the C ABI is a path string + a function pointer, so you can hand-register
instead; `docs/C_API_GUIDE.md` shows that form.

See `docs/C_API_GUIDE.md` (repo) / `sdk/docs/C_API_GUIDE.md` (SDK bundle)
for the full plain-C guide, and the C++ example for the richer showcase
(streaming, health, deferred completion) ‚Äî `../ble` in the repo,
`../example` in the SDK bundle.
